# Per-stage screenshot capture for ScenarioScript pages

## File outputs

```
public/script-thumbs/
  {scenarioId}.png             ← hero (= intro frame)
  {scenarioId}__detect.png
  {scenarioId}__observe.png
  {scenarioId}__hypothesize.png
  {scenarioId}__plan.png
  {scenarioId}__act.png
  {scenarioId}__verify.png
  {scenarioId}__resolved.png
```

The hero (`{id}.png`) is just a copy of `__detect.png` (or the first stage that exists) so the existing `ScreenshotCard` continues to work unchanged.

Total disk: \~600 PNGs × \~50 kB = \~30 MB (1280×720, PNG-optimised). Acceptable for the public/ folder.

---

## Approach

### 1. Add a tiny URL-driven auto-runner inside the app

A new `<ScenarioAutoRunner />` mounted high in the tree. On every route change it reads `?run=<scenarioId>&stage=<stage>` from the URL.

Behaviour:

- `?run=<id>` only → call `selectIncident(id)`, navigate to the scenario's home section route, do not auto-play.
- `?run=<id>&stage=<key>` → call `selectIncident(id)`, navigate, then once the scenario timeline starts, **fast-forward** firedEvents until the requested stage's first event has fired. Then **freeze** (no further ticks). Posts a `window.__captureReady = true` flag for Playwright to wait on.
- `&clean=1` → hides the Narrator/Sidebar chrome via a `body.capture-mode` class so screenshots show only the stage.

This is \~80 LOC in a single new component + a small CSS rule.

### 2. Add Playwright capture script

`scripts/capture-thumbs.mjs`:

- Imports the scenario list at runtime by reading `src/data/sectionScenarios.ts` and `src/data/nocIncidents.ts` via Vite's compiled output, OR (simpler) via a static JSON manifest committed under `scripts/scenarios.manifest.json` regenerated by a `scripts/build-manifest.mjs` step.
  - To avoid TS-import gymnastics in node, use **Approach B**: a tiny `scripts/build-manifest.mjs` that uses esbuild to compile `src/data/sectionScenarios.ts` + `src/data/nocIncidents.ts` to ESM and emits `scripts/scenarios.manifest.json` with `[{id, sectionId, stages: ['detect','observe',...] }]`.

- For each scenario × each stage:

  1. `page.goto(http://localhost:5173/?run={id}&stage={stage}&clean=1)`
  2. wait for `window.__captureReady`
  3. wait `200ms` for animations to settle
  4. screenshot `1280×720` → `public/script-thumbs/{id}__{stage}.png`

- After all scenarios done, copy each `__detect.png` (or first available stage) to `{id}.png` for the hero shot.

- Concurrency 4 to keep it fast (\~3-4 minutes total for 600 captures).

### 3. Update ScenarioScript.tsx

- Each `Block` now accepts an optional `stageKey` prop.
- When provided, the block renders a `<ScreenshotCard scenarioId={id} stage={stageKey}>` *inside* the block, after the spoken text, before the live event log.
- `ScreenshotCard` extended: optional `stage?: string` prop — when set, src becomes `public/script-thumbs/{id}__{stage}.png`; when absent, falls back to `{id}.png`; final fallback is the gradient placeholder.

### 4. npm scripts

```
"build-manifest": "node scripts/build-manifest.mjs",
"capture-thumbs": "npm run build-manifest && node scripts/capture-thumbs.mjs",
"capture-thumbs:hero-only": "node scripts/capture-thumbs.mjs --hero-only"
```

The `--hero-only` flag captures one PNG per scenario (the intro frame) for the lighter Option-2 outcome if you want fast feedback before running the full Option-3 pass.

### 5. Dependencies

```
npm i -D playwright @playwright/test esbuild
npx playwright install chromium
```

Chromium-only (no Firefox/WebKit) keeps the install \~150 MB.

## Run flow

```
# Terminal 1 — dev server
npm run dev

# Terminal 2 — capture
npm run capture-thumbs
```

The script auto-detects whether the dev server is up; if not, it spawns it on a random port and shuts it down at the end. \~3-5 min.

To capture only a single scenario for iteration:

```
node scripts/capture-thumbs.mjs --only=cic-manchester-churn
```

To re-capture only a single stage across all scenarios:

```
node scripts/capture-thumbs.mjs --stage=resolved
```
